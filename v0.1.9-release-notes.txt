Aqwel-Aion v0.1.9 — What changed and why (plain explanation)
================================================================

This file summarizes additions and updates in release 0.1.9: module names,
intent, and how the pieces fit together. It does not list repository folder
trees; see README.md for layout and installation.

----------------------------------------------------------------------
VERSION AND PACKAGING
----------------------------------------------------------------------

Version was bumped to 0.1.9 in pyproject.toml, setup.py, main.py, and
aion/__init__ so PyPI installs, editable installs, and "import aion;
print(aion.__version__)" all report the same number. That keeps releases
traceable for users and for bug reports.

----------------------------------------------------------------------
NEW OR HIGHLIGHTED MODULES
----------------------------------------------------------------------

1) aion.datasets
   Why added:
   Research and training scripts repeatedly need the same small patterns:
   read text/CSV/JSONL, stream large files without loading everything into
   memory, batch items for training loops, and sanity-check that each record
   has required keys (e.g. "text" and "label").

   What it provides (names users import from aion.datasets):
   - load_text, iter_text_lines — plain text files
   - load_csv, iter_csv_rows — tabular CSV
   - load_jsonl, load_json_lines_file, iter_jsonl — JSON Lines
   - batch_processor — fixed-size batches with clear error on invalid batch size
   - validate_schema — lightweight dict/key checks for pipeline safety

   How well it works:
   These are intentionally small, dependency-light helpers built on the
   standard library plus normal file I/O. They fit between ad-hoc scripts and
   heavier frameworks (pandas, Hugging Face datasets). They do not replace
   aion.former.datasets, which stays focused on transformer tokenization and
   sliding-window text for Former.

2) aion.io
   Why added:
   High-level file helpers (aion.files) are useful for day-to-day CRUD;
   pipelines also need predictable low-level behavior: streaming reads for big
   logs, atomic writes so a crash mid-write does not leave a half-written
   config, and checksums for reproducibility and integrity checks.

   What it provides:
   - iter_lines, read_chunks — streaming consumption
   - atomic_write, atomic_write_bytes, save_automatically — safer persistence
   - file_sha256, verify_sha256 — digest and verification

   How well it works:
   The API is orthogonal to aion.files: use io for correctness and streaming
   patterns; use files for higher-level operations. All of this stays usable
   in headless and CI environments without extra services.

3) aion.providers
   Why added:
   Many projects call OpenAI, Gemini, Anthropic, or an OpenAI-compatible
   server. A small, consistent interface reduces copy-paste and keeps
   environment-variable conventions documented in one place.

   What it provides:
   - OpenAIProvider, GeminiProvider, AnthropicProvider,
     OpenAICompatibleProvider
   - ChatProvider protocol, ChatMessage type, ProviderError
   - create_provider, supported_providers — discovery and factory-style setup

   How well it works:
   Implementations are thin clients over vendor HTTP APIs; they depend on
   network access and valid API keys (see each class and aion.providers
   package docstring for env vars). They are suitable for scripts and
   services that already manage secrets; they are not a full agent framework.

4) aion._core (user-facing: fast_* and using_native_extension)
   Why added:
   Some numerical hot paths benefit from a native implementation when a C++
   toolchain and pybind11 are available. The rest of the library should keep
   working everywhere with NumPy-only fallbacks.

   What it provides:
   Exported on the main package: fast_sum, fast_dot, fast_norm2, fast_mean,
   fast_variance, fast_argmax, fast_relu, fast_softmax, fast_cumsum,
   fast_matrix_vector_mul, fast_lower_bound, and using_native_extension.

   How well it works:
   If aion._aion_core is built from src/aion_core.cpp, calls can use the
   native path; otherwise _core transparently uses NumPy. That gives a single
   import surface for notebooks and libraries without conditional imports
   scattered through user code.

----------------------------------------------------------------------
TESTS
----------------------------------------------------------------------

tests/test_smoke.py (run with pytest after pip install -e ".[dev]")
   Why added:
   Smoke tests catch regressions in small, stable utilities (datasets
   batching, schema validation, and related imports) without requiring GPUs or
   paid APIs.

   How well it works:
   They validate that the package imports and core helpers behave as expected
   in a minimal environment; they are not a full integration or coverage
   suite for every optional extra.

----------------------------------------------------------------------
DOCUMENTATION (README AND VERSION STORY)
----------------------------------------------------------------------

README was expanded for 0.1.9 to describe the new subpackages, optional
native numerics, how to run tests, and usage examples (datasets, io,
providers, fast_*). The goal is that a new contributor understands what
each area is for without reading the whole codebase first.

----------------------------------------------------------------------
DESIGN TAKEAWAY
----------------------------------------------------------------------

0.1.9 adds narrow, composable layers: data loading (datasets), safe I/O (io),
remote LLM calls (providers), and optional speedups (_core). Together they
support typical research workflows—prepare data, write artifacts safely,
call models, optionally accelerate numerics—while keeping heavy stacks
still behind optional extras like [ai], [full], and [dev].
